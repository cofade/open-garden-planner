## 11.4 Known Development Pitfalls

Hard-won lessons from implementation. Read these before modifying the related subsystems.

- **Sidebar list panels must store item *ids*, not live `QGraphicsItem` refs, and defer the selection** (issue #212): `PlantSearchPanel` ("Find Plants") used to cache the live `QGraphicsItem` in each row's `UserRole` and call `setSelected(True)` on it inline. Two problems compounded into "selection works only sometimes": (1) the cached reference goes stale after undo/redo/delete/reparent â€” `setSelected` then silently no-ops on a detached item; (2) `QGraphicsScene.changed` (very chatty â€” fires on every repaint, including the repaint caused by selecting) was wired straight to a full `refresh_plant_list()` that `clear()`s and rebuilds every row, so a refresh could destroy the row the user was mid-clicking. Fix: store only the stable `item_id`, resolve the live item at click time via `scene.find_item_by_id(item_id)` (returns `None` â†’ refresh the list and bail, no crash); defer the scene mutation with `QTimer.singleShot(0, ...)` so it runs after the `QListWidget` finishes its own click processing (same pattern as `_on_companion_highlight_species`); and debounce the `scene.changed` â†’ refresh through a single-shot `QTimer` (~150 ms) so the list isn't torn down on every repaint. Single-click uses `view.ensureVisible()` (scroll only if off-screen); double-click uses `view.centerOn()`. General rule: any sidebar list that drives canvas selection must key on ids + defer, never hold graphics-item references across scene mutations.

  **Follow-up (#212 manual-test round 2): debouncing `scene.changed` is NOT enough â€” the refresh must also skip the rebuild when nothing changed.** The 150 ms debounce only *coalesces* a burst; it still fires once the burst settles and still did a destructive `results_list.clear()` + recreate-every-row. Because *selecting* a plant triggers its own burst (`selectionChanged` â†’ `_update_spacing_overlaps`/`_update_companion_highlights` mutate item visuals â†’ repaint â†’ more `scene.changed`), every selection ended in a full list rebuild â€” which (a) reset the list's scroll position (symptom: "list jumps back to the top a moment after I stop scrolling" with >5 plants) and (b) destroyed the row the user just clicked, racing the deferred `_apply_selection` (symptom: "single-click sometimes needs a second/third click"). Fix: `_update_results_display` computes a **signature** of the visible rows (ordered tuple of `(item_id, name, species, plant_type)` after sort+search+type-filter) and **early-returns when it equals the last rendered signature** â€” so a no-op refresh touches neither scroll nor selection. On a real rebuild it captures+restores the scroll value and re-selects the prior row by `item_id` (scroll preserved on scene-driven refreshes; reset to top on user search/filter via a `from_user_input` flag). `family` is excluded from the signature (search-only, never displayed; any family change that alters results already changes membership). Lesson: a debounce limits *how often* you rebuild, not *whether* you needed to â€” a list driven by the repaint firehose must also be **idempotent** (diff before rebuild) or it will fight the user's scroll and clicks. (The underlying "rebuild the world on every repaint" smell is the same one tracked for the properties panel in #206; the signature guard is a local band-aid, not a structural fix â€” the firehose still re-queries the whole scene every tick.) Also hardened: row widgets installed via `setItemWidget` set `WA_TransparentForMouseEvents` so a press always hit-tests to the list row regardless of the row widget's child layout.

- **Writing `metadata["plant_species"]` does not repaint the canvas, and a manual `spacing_radius_cm` override masks the database value** (issue #213): the spacing circle's radius comes from `effective_spacing_radius()`, which returns the user override *first* and only falls back to `max_spread_cm/2` from `metadata["plant_species"]`. So a direct `item.metadata["plant_species"] = â€¦` (as several `plant_database_panel` paths did) updates the data model but (a) never triggers `prepareGeometryChange()`/`update()`, so the spacing circle keeps its old size on screen, and (b) is silently overridden on screen by any manual spacing the user set earlier. Contract: route species *assignment* through `PlantDatabasePanel._apply_species_to_item()`, which builds a single undoable `ApplySpeciesCommand` (sets metadata **and**, on *Apply database values*, resizes the drawn footprint to `max_spread_cm` via `CircleItem.set_radius_centered()`, then repaints). The footprint resize is **silent**; the only thing that prompts is a manual `spacing_radius_cm` override that *conflicts* with the database value (Apply â†’ resize + clear override; Keep â†’ leave both). Do **not** wrap live per-keystroke field edits (`_on_field_changed`) in that command â€” that recreates the per-keystroke undo-churn fixed in #210; there, write metadata directly and just call `prepareGeometryChange()`/`update()` for the repaint.

  **Manual-test follow-up (#213): refreshing only the spacing ring was not enough.** The spacing ring is only painted when the item is selected **and** `spacing_r > footprint_radius` (`circle_item.py`); when a generic plant is placed at a footprint larger than the assigned species' spread (the common case â€” a 5 m placeholder getting a 1.5 m shrub), the ring falls *inside* the footprint and is never drawn, so "nothing resizes on screen" despite the panel showing the new values. The issue explicitly lists **Diameter** among the properties that must update, so *Apply database values* now resizes the **drawn footprint** itself (diameter == `max_spread_cm`, centre fixed) rather than only the ring. Lesson: the visible artefact a user expects to change is the footprint circle, not the (often hidden) spacing ring â€” reconcile the thing on screen.

  **Second trap (senior review): programmatic resize must keep the rotation pivot on the centre.** A `CircleItem`'s saved position is `pos + center` (see `core/project.py` serialization), with rotation stored as a *separate* angle pivoting on the centre â€” i.e. the code relies on the invariant `transformOriginPoint == center == rect.center()`. A resize that only keeps the *visual* centre fixed (via `mapToScene` compensation) but forgets to move `transformOriginPoint` to the new centre will save a **displaced** `center_x/center_y` for any **rotated** plant (and make the next rotation jump). It looks correct on screen immediately, so an unrotated test passes green. `CircleItem.set_radius_centered()` therefore calls `setTransformOriginPoint(new_center)` and repositions so `pos + center` stays put; `test_apply_keeps_rotated_plant_centered` pins it. General rule for this codebase: any geometry mutation on a rotatable item must preserve `transformOriginPoint == rect.center()`, and resize-under-rotation needs an explicit rotated-item test.

- **Release workflow race condition with chore commits**: After merging a feature PR, two chore commits are pushed (sync version + mark progress). These land ~37s after the PR merge but the Release workflow building the new tag takes ~2m50s. The chore-commit Release runs start while the tag doesn't exist yet, compute a stale version (e.g., `v1.8.4` instead of `v1.9.2`), and fail with "release with the same tag name already exists". Fixed by adding `if: "!startsWith(github.event.head_commit.message, 'chore:')"` to the release job, which skips the workflow for chore commits.

- **Anchor index on same-type anchors**: When multiple anchors share the same `AnchorType` (e.g. rectangle corners are all `CORNER`, polygon vertices are all `CORNER`, polyline vertices are all `ENDPOINT`), each must have a unique `anchor_index` in `get_anchor_points()`. Without it, `DimensionLineManager._resolve_anchor_position()` falls back to type-only matching and picks the first anchor. Always pass `anchor_index=i` when creating `AnchorPoint` for same-type anchors.

- **Dimension line updates after undo/redo**: `CommandManager.command_executed` only fires on `execute()`, NOT on `undo()`/`redo()`. Dimension line updates must also be connected to `can_undo_changed`/`can_redo_changed` signals.

- **3-anchor constraints not solved on add**: `_compute_constraint_solve_moves()` in `canvas_view.py` collects `constrained_ids` from `anchor_a` and `anchor_b` only. Any constraint with a third anchor (`anchor_c`, e.g. ANGLE) must also add `anchor_c.item_id` here, otherwise the third item is absent from `item_positions` and the solver cannot move it â€” showing as red/violated until the user manually drags an object.

- **Live-drag projection tolerance must be sub-pixel** (PR #169 follow-up): `ConstraintGraph.project_to_feasible()` calls `newton_refine()` with a `tol` parameter that gates an early-return *before* projection: `if max_err <= tol: return`. The starting state has `vertex_pos[moving] = desired_scene_pos` (the cursor). At cm-scale `tol` (e.g. 0.5 cm), the cursor near a feasible point falls inside the slack band on most frames of a near-stationary drag, so the function returns the cursor **unchanged** â€” the moving vertex slips with the cursor up to that band each frame. On a fully-constrained chain (EDGE_LENGTH on every adjacent pair), the slip looks like the entire polyline rigidly translating because the user can only see the moving vertex's position in scene coords. Fix: `project_to_feasible` defaults to `tolerance=1e-4` (sub-render-precision); `solve_anchored` and other full-graph callers retain their cm-scale tolerances. Defensive companion: `newton_refine` now also calls `write_x(x)` on the early-return path so callers always observe a consistent state.

- **Canvas Y-axis flip**: The view applies `scale(zoom, -zoom)` so **positive scene Y is visually upward** on canvas (CAD-style, origin bottom-left). When computing directional offsets from user-facing angles (e.g. linear array), negate `dy`: `dy = -spacing * sin(angle_rad)` so that 0Â°=right, 90Â°=down, 180Â°=left, 270Â°=up matches screen-space intuition. The canvas rect in scene coords is `QRectF(0, 0, width_cm, height_cm)` accessed via `self._canvas_scene.canvas_rect`.

- **EDGE_* anchor type instability on polygons/polylines** (TD-007): `_polygon_anchors()` in `measure_snapper.py` classifies each edge as `EDGE_TOP/BOTTOM/LEFT/RIGHT` based on its **current** dominant axis (horizontal vs vertical). This is a dynamic, geometry-dependent label. When a vertex is dragged far enough to flip an edge's dominant axis (e.g. a nearly-horizontal edge becomes nearly-vertical), the freshly-computed anchor type differs from the value stored in the constraint record. `_resolve_anchor_position()` in `dimension_lines.py` then fails on its `(type AND index)` match and falls through to type-only matching, snapping the constraint indicator to the wrong edge. Current workaround: an EDGE_*-aware index-only match block at the top of `_resolve_anchor_position` catches these mismatches before the fallback fires. Long-term fix (TD-007): replace all four `EDGE_*` types with a single `AnchorType.EDGE_MIDPOINT` and use `anchor_index` as the sole edge identity. The edge is always identified by its start-vertex index, not by axis classification, making the anchor stable across all vertex moves.

- **`scene.render()` with negative-height target rect is empty (PyQt6)**: `QRectF(0, H, W, -H).isEmpty()` returns `True`. Passing this as the target to `scene.render()` clips to an empty region â€” nothing is painted. The correct Y-flip technique is `painter.save(); painter.translate(0, H_PIXELS); painter.scale(1.0, -1.0); scene.render(painter, QRectF(0, 0, W, H), source_rect); painter.restore()`. `H_PIXELS` must be the **image height in pixels**, not the canvas height in cm.

- **`ItemIgnoresTransformations` items render at fixed device-pixel size in `scene.render()`**: When rendering via `scene.render()` (not through a QGraphicsView), `ItemIgnoresTransformations` items render at their natural pixel size regardless of the sourceâ†’target scale. For a large scene (e.g. 5300 cm) rendered into a small image (774 px), a 10 pt text item can appear as 80+ cm on canvas. Fix: call `ExportService._prepare_text_for_export(scene, scale, dpi)` before rendering and `_restore_text_after_export()` after. The scale = `output_width_cm / canvas_width_cm`.

- **Qt silently drops the mouse grab on `ItemIgnoresTransformations` handles between events (PyQt6)** (#193 follow-up): A handle item that sets `ItemIgnoresTransformations` (all our drag handles â€” `VertexHandle`, `ResizeHandle`, `RotationHandle`, `RectCornerHandle`, `MidpointHandle`, `CurveControlHandle`) grabs the mouse in its `mousePressEvent`, but Qt loses that grab before the first `mouseMoveEvent` â€” so the handle receives the press and then *nothing*, and the drag appears inert (symptom: "handle shows but won't move"). **Root cause**: PyQt6 quirk with `ItemIgnoresTransformations` + child-item mouse grab. **Fix/contract**: `CanvasView` re-establishes the grab each `mouseMove`/`mouseRelease` (`if self._active_drag_handle is not None and scene().mouseGrabberItem() is None: self._active_drag_handle.grabMouse()`), but only for handle types it recognises â€” **every new such handle MUST be added to the `isinstance(grabber, (...))` allow-list in `CanvasView.mousePressEvent`** (grep `_active_drag_handle`). A faithful copy of an existing handle is still dead until the view knows its type. **Testing note**: a hand-built `QMouseEvent` is *not* delivered to an `ItemIgnoresTransformations` child â€” use `QTest.mousePress/Move/Release` on `view.viewport()` (after `centerOn` the target) for real handle-drag tests. See ADR-025 and the debug-verbose case study.

- **QPdfWriter must use `setResolution(72)` before `setPageLayout()`**: Default QPdfWriter resolution is ~1200 DPI. `painter.viewport()` returns device-pixel dimensions, so all layout math in points (1/72 inch) lands in the top-left corner. Adding `writer.setResolution(72)` makes device pixels equal PDF points and layout coordinates match.

- **`scene.render()` directly on QPdfWriter painter is unreliable (Y-flip + margins)**: QPdfWriter's painter has a non-identity initial transform from margin handling. The painter pre-flip formula `translate(0, cr.top + cr.bottom); scale(1,-1)` is mathematically correct in isolation but produces wrong output because the initial transform shifts the baseline. Fix: render the scene into a temporary `QImage` (where pre-flip is reliable) then embed with `painter.drawImage(content_rect, img)`.

- **Plant items store center in `boundingRect()`, not in `pos()`**: OGP `CircleItem` and similar types define their geometry in local coords with `pos()` always at (0, 0). `item.pos()` and `item.scenePos()` both return (0, 0). To get the visual center in scene coordinates: `item.mapToScene(item.boundingRect().center())`.

- **SVG pattern tiles appear Y-inverted under painter Y-flip**: Qt's `QSvgGenerator` records `<pattern>` elements with `patternUnits="userSpaceOnUse"`. When the scene is rendered with a `scale(1,-1)` painter transform, each pattern tile image is stored un-flipped, but renders upside-down inside the Y-flipped coordinate space. Fix: post-process the SVG after `painter.end()` and add `patternTransform="matrix(1,0,0,-1,0,{tile_height})"` to each `<pattern>` element. See `ExportService._fix_svg_pattern_yflip()`.

- **`QSvgGenerator` does not serialize painter clip regions â€” texture fills bleed across the canvas**: For texture-filled shapes, Qt emits a "shadow" group containing the polygon `<path>` (`fill="#000000"`) followed by a texture group `<g fill="url(#tex)"><rect x="..." y="..." width="..." height="..."/></g>`. The rect is the *painter's clip bounding rect*, not the polygon shape â€” and the painter's `setClipPath()` is **never** serialized as `<clipPath>`. In SVG viewers (browsers), the unconstrained rect washes across the whole canvas (often as a brownish overlay if it is a roof-tile texture). PNG export is unaffected because Qt's native renderer applies the clip directly. Fix: post-process the SVG (`ExportService._fix_svg_qt_texture_clipping()`) â€” pair each non-empty shadow group with the next non-empty texture group in document order, build a `<clipPath>` from the shadow's `<path>` (preserving its `transform`), and wrap the texture group with `clip-path="url(#...)"`. Pairing must be 1:1 with a `used_textures` set; forward-window scans match the same texture from multiple shadows and corrupt the XML tree. Validate SVGs in a real browser (`scripts/svg_preview.py` uses Edge headless) â€” `QSvgRenderer` is too forgiving and hides this bug.

- **DXF Y-axis: scene Y is already Y-up, no negation needed**: OGP scene uses Y-up (scene Y=0 = visual bottom). DXF also uses Y-up. The formula `dxf_y = canvas_h - scene_y` double-flips. Correct mapping: `dxf_y = scene_y`. Similarly for import: `scene_y = dxf_y * scale` (no negation).

- **`QGraphicsScene.changed` only fires on geometry/visibility, not on Python attribute mutations** (issue #173): The 500 ms debounce timer in `CanvasView` that drives `_update_soil_mismatches` is wired to `scene.changed`. Mutating `parent_bed_id` / `_child_item_ids` on a `GardenItemMixin` instance is a plain Python attribute write â€” no signal, no scene invalidation, no debounce trigger. Any code path that re-parents a plant (drag-and-drop, properties-panel "Unlink", paste, duplicate, future callers) must explicitly call `trigger_soil_mismatch_refresh(scene)` (in `commands.py`), or the warning borders go stale until the next unrelated scene change. Funnel attach/detach through `SetParentBedCommand` whenever possible â€” it triggers the refresh itself.

- **Plants stack behind beds when both share a layer's default z**: A new `CircleItem` and a new `RectangleItem` placed on the same layer both get `z_order * 100` from the layer (typically `0`). With equal z-values, Qt stacks by add-order. So a plant drawn *before* its bed renders behind it, even when correctly attached as a child. Fix is to elevate the plant above the bed at every attach site: `ensure_z_above_parent(plant, bed)` in `commands.py`. Already wired into `_auto_parent_plant`, `SetParentBedCommand`, `DeleteItemsCommand.undo`, paste, and duplicate. Any new path that establishes a plant-bed parent link must also call this helper.

- **Context-menu wiring must cover all bed-shape item types**: OGP has four independently-implemented item classes for beds (`RectangleItem`, `PolygonItem`, `EllipseItem`, `CircleItem`). Every new context-menu action added to one must be explicitly added to all others â€” there is no shared base-class hook for context menus. US-12.7 initially missed `PolygonItem` and `EllipseItem`; both were discovered missing during manual testing.

- **Two-tab dialog: OK must not trigger entry-tab validation when user is on History tab**: Dialogs with an Entry tab (add a new record) and a History tab (view/edit past records) share one OK button. If the user is on the History tab and clicks OK to dismiss the dialog, any validation on the Entry tab (e.g. "empty name" warning) fires incorrectly against an empty form. Guard with `self._tabs.currentIndex() == 0` before running entry-tab validation.

- **Two-tab dialog: caller must not create a new record when user clicked OK from History tab**: When the outer dialog returns `Accepted`, the caller (`_open_pest_log_dialog` in `application.py`) must not unconditionally call `AddPestLogCommand`. If the user was on the History tab, edits were already committed via `EditPestLogCommand` inside the dialog; the outer accept is just a close. Add a `has_new_entry` property to the dialog (set in `_on_accept` based on which tab is active) and gate the add-command on it. Same pattern applies to the soil-test dialog.

- **`mousePressEvent` monkey-patch must return None, not the result of `openUrl()`**: Overriding `mousePressEvent` on a `QLabel` instance via assignment works, but the assigned callable must return `None` (C++ void). `QDesktopServices.openUrl()` returns `bool`. A lambda `lambda _e: QDesktopServices.openUrl(...)` forwards that bool, and PyQt6/SIP raises `TypeError: invalid argument to sipBadCatcherResult()` on return. Fix: use a named inner function (implicit `None` return) or add `, None)[1]` to discard the bool.
- **`ThemeColors.get_colors(ThemeMode.SYSTEM)` ignores the user's theme choice**: `apply_theme()` only writes a stylesheet â€” it does *not* mutate `QApplication.palette()`. So `detect_system_theme()` (called via `ThemeMode.SYSTEM`) probes the OS palette regardless of what the user picked in Settings â†’ Appearance. A user with `theme_mode=LIGHT` running on a dark-default OS will get the dark-mode palette dict for any code that asks "what colors am I drawing on top of?". Fix: always pass the user's preference â€” `ThemeColors.get_colors(get_settings().theme_mode)`. Only `theme.py` should call `get_colors()` with `ThemeMode.SYSTEM`; every other consumer must read `get_settings().theme_mode`.
- **Excel reads UTF-8 CSV as Latin-1 unless a BOM is present**: A bare `encoding="utf-8"` write produces correct bytes but Excel on Windows still mojibakes German Umlauts (`TÃ¼te` â†’ `TÃƒÂ¼te`). Use `encoding="utf-8-sig"` for any CSV the user is expected to open in Excel. Python's own `csv` module strips the BOM transparently on read with either `utf-8` or `utf-8-sig`, so existing tests don't need to change. The BOM byte sequence is `b"\xef\xbb\xbf"` if you need to assert it directly.
- **`ShoppingListItem.id` must encode the display unit when units auto-promote**: `ShoppingListService._collect_materials` promotes amendment quantities from g to kg once the cross-bed total crosses 1000 g. Because user-entered prices in `ProjectData.shopping_list_prices` are keyed by `id`, the id must capture the unit (`amendment:<aid>:g` vs `amendment:<aid>:kg`) â€” otherwise a price entered when the row was 800 g silently re-binds to the same row when it later weighs 1.2 kg, multiplying the apparent total by 1000Ã—. Use the locale-stable suffix (`g`, `kg`) â€” never the translated `_tr()` string â€” so the id is identical across English/German runs.

- **Bed-only features must NOT be hand-coded into each shape item**: Bed-capable shapes are four classes (`RectangleItem`, `PolygonItem`, `EllipseItem`, `CircleItem`). When a new bed-only action (soil test, pest log, succession plan, future ones) is added to only one shape's `contextMenuEvent`, the feature is invisible on the other three. Recurred twice: pest log missing from PolygonItem/EllipseItem until #173; succession plan missing from all three non-rectangle shapes until the US-12.8 post-bug fix. Mitigation: bed actions are now built and dispatched centrally via `GardenItemMixin.build_bed_context_menu` + `dispatch_bed_action`, gated by a parametrised regression test (`tests/integration/test_bed_context_menu.py`) that iterates over all four shape classes. See ADR-017 and Â§ 8.14. **When adding a new bed feature, extend the parametrised test with one new `assert actions.<new_field> is not None` line per shape â€” that single addition is what blocks future regressions.**

- **`painter.drawText()` inside `QGraphicsItem.paint()` renders upside-down on the Y-flipped canvas**: The view applies `scale(zoom, -zoom)` for the CAD-style bottom-left origin (ADR-002). Text drawn directly inside a `paint()` method inherits this flip and appears mirrored. Symptoms: badge text reads as mirror image; positioning using `rect.bottom()` lands at visual *top* and vice versa. Fix: use a child `QGraphicsSimpleTextItem` (or a custom `QGraphicsItem` subclass) with the `ItemIgnoresTransformations` flag â€” Qt then draws the item in its own non-flipped pixel-coordinate system. Examples: bed name labels (`_label_item` in `garden_item.py`), succession badge (`SuccessionBadgeItem` in `garden_item.py`), compare-mode overlay labels (`canvas_scene.py`). Anchor at `rect.top()` (smallest scene-Y = visual bottom after flip) and extend the bounding rect upward (negative local Y) so the badge sits inside the bed.

For multi-file bulk replacements use Python, `sed -i.bak`, or `perl -i -pe` â€” anything but PowerShell. - **`test_german_ts_has_no_unfinished` does NOT catch hardcoded English strings in source code**: The i18n test only verifies that every string *already extracted into the .ts file* has a German translation; it cannot see strings that bypass `tr()` / `QT_TR_NOOP` / `QCoreApplication.translate(...)` entirely. Symptom: registering a German translation in `scripts/fill_translations.py` doesn't make the string translate in the running app, because the source-code call site still writes a plain Python f-string / dict literal that pylupdate6 never picks up. Concrete failures in this codebase: module-level `_SEGMENT_LABELS = {"early_spring": "Early Spring", ...}` dict was looked up directly with `_SEGMENT_LABELS[key]` in three places â€” the strings were registered in fill_translations.py but the dialog displayed the raw English. The fix pattern: wrap dict literals with `QT_TR_NOOP` and translate at lookup time, OR add a thin helper like `_segment_label(key)` that calls `QCoreApplication.translate("Context", english)`. Also: `f"{a} overlaps {b}: antagonist"` looks like a translatable sentence but never reaches Qt Linguist because `tr()` is not on the path; use `self.tr("...").format(...)` instead. Mitigation: `tests/unit/test_i18n.py::TestNoHardcodedEnglish` greps a curated list of suspicious phrases (e.g. ` overlaps `, ` antagonist`, `Early Spring`, `Late Spring`) against the source tree â€” if any match is outside a `tr()`/`QT_TR_NOOP`/`translate()` call, the test fails. The list grows whenever a hardcoded string slips through user-visible testing.

- **Layer subsystem: panel is a pure view; ALL mutations go through undoable commands** (issue #201 + layer-undo fix): historically `LayersPanel` mutated layers directly and `panel._layers` *aliased* `scene._layers` (the scene hands its live list to `panel.set_layers()`), so panel-side inserts mutated the scene before any signal fired â€” and none of it was undoable (Ctrl+Z silently ignored every layer action). Both are fixed: `LayersPanel.set_layers` now stores a **defensive copy** (`list(layers)`), and every layer operation (add, delete, rename, reorder, visibility, lock, opacity) routes panel request-signal â†’ `GardenPlannerApp` handler â†’ `CommandManager.execute(<LayerCommand>)` â†’ scene mutation â†’ `layers_changed` â†’ `panel.set_layers` rebuild. Rules for any new code touching this subsystem: (1) the panel must NEVER mutate `Layer` objects or its `_layers` list â€” emit a request signal and let a command in `core/commands.py` (`AddLayerCommand`, `DeleteLayerCommand`, `RenameLayerCommand`, `ReorderLayersCommand`, `SetLayerPropertyCommand`) do the work; the rebuild repaints the widgets. Emit as the **last** statement of a `LayerListItem` handler â€” the synchronous command â†’ rebuild deletes the emitting widget. (2) `_refresh_list` restores selection by **layer id**, never row index â€” a top-insert shifts every row down one. (3) Undo of add/delete restores the previously active layer by calling `set_active_layer` **last**, after `layers_changed.emit()` â€” the panel rebuild's fallback `setCurrentRow(0)` fires first and would otherwise win; the scene's `active_layer_changed â†’ panel.select_layer` connection (signal-blocked) re-syncs the visible selection. (4) Opacity slider drags coalesce into one undo step: `valueChanged` while `isSliderDown()` drives the non-undoable `scene.preview_layer_opacity()` (no `layers_changed`, so no rebuild storm per tick); `sliderReleased` commits one `SetLayerPropertyCommand` carrying the opacity snapshotted at `sliderPressed`. (5) `CanvasScene.add_layer()` *appends* (bottom) and is for bulk/import only (DXF); `CanvasScene.remove_layer()` is likewise a non-undoable primitive â€” neither is the user-facing path. (6) Command `description` strings are user-visible (status bar "Undo: â€¦") and use `QCoreApplication.translate("Commands", â€¦)` + registration in `scripts/fill_translations.py`. (As of issue #209 **all** command descriptions â€” not just the layer ones â€” use this pattern; the former English f-strings have been migrated, including the caller-supplied descriptions of `AlignItemsCommand`/`MultiVertexMoveCommand`, which are wrapped at their call sites in `canvas_view.py`/`properties_panel.py`.)

- **`QLineEdit` ignores Return â€” Enter bubbles to the canvas and finalizes the active tool** (Package A US-A4): `QLineEdit.keyPressEvent` calls `event.ignore()` for `Key_Return`/`Key_Enter` so a parent (typically a dialog's default button) can pick it up. Inside the Dynamic Input overlay this means the Enter that committed a typed coordinate then propagated to `CanvasView.keyPressEvent`, which delegated to `PolylineTool.key_press` and triggered `_finish_polyline()` whenever the tool already had â‰¥2 vertices. Symptom: every typed `@500,0` Enter placed the vertex correctly, then immediately closed the polyline â€” you could only ever draw one segment via the overlay. Fix: `_DynamicLineEdit.keyPressEvent` now intercepts Return/Enter explicitly, drives commit (or empty-Enter finalize) via the overlay, and calls `event.accept()` so the Enter never reaches the canvas. Same trap will bite any future QLineEdit-derived widget hosted inside a `QGraphicsView` viewport â€” if the parent acts on Return, intercept it at the line edit.
- **Dynamic Input overlay must freeze while the user is typing** (Package A US-A4): The overlay floats next to the cursor with a fixed `(+18, -42)` offset and was originally re-positioned on every `mouseMoveEvent`. The user could never reach it with the mouse â€” every step toward the fields shifted them further along the cursor vector. Equally bad, naked keystrokes on the canvas didn't reach the overlay because canvas focus owned the keyboard. Fix: `DynamicInputOverlay.is_capturing_input()` reports True iff either field has non-empty text; `CanvasView._update_dynamic_overlay` skips `show_near` while that is True; `CanvasView.keyPressEvent` routes bare coord characters (`0-9 . , ; @ < + - space`) into the overlay's distance edit via `forward_keystroke` before the tool-shortcut dispatch. After commit, fields are cleared (via `_on_buffer_text_changed("")`) so the overlay flips back to tracking. Pitfall variant: do NOT base `is_capturing_input` on focus alone â€” focus may legitimately sit on the field after commit while the buffer is empty, and the overlay should resume tracking in that state.

- **Grid snap silently overrides anchor snap unless `CanvasView.snap_point` short-circuits** (Package A, ADR-020/021): Every drawing tool calls `self._view.snap_point(scene_pos)` inside its own `mouse_press`/`mouse_move`/`mouse_release`. The `CanvasView` dispatcher (`_maybe_apply_anchor_snap`) snaps the scene_pos to a midpoint/intersection/endpoint candidate *before* dispatching to the tool â€” but the tool's own `snap_point` call then rounds that anchor-snapped coordinate back to the nearest grid line (e.g. an edge midpoint at (175, 100) becomes (200, 100) on the default 50 cm grid). With grid snap on (the default), the new Package A snap modes silently lose. Symptom: midpoint/intersection glyph appears on hover but the committed vertex lands on a grid line, not the visible anchor. Root cause: anchor-snap and grid-snap are independently applied by two unrelated layers without composition rules. Fix: `CanvasView.snap_point()` short-circuits the grid rounding step when `self._current_snap is not None`. Anchor snap always wins on the click path; the bounds clamp still runs. Regression test: `tests/integration/test_anchor_snap.py::test_grid_snap_does_not_override_anchor_snap` (uses a 350Ã—100 rect with a midpoint at (175, 100) â€” deliberately not on the 50 cm grid). The single short-circuit site keeps this an explicit invariant rather than a side-effect of call ordering â€” if you find yourself adding a third snap layer, add an explicit composition rule before merging.

- **Never default a file dialog into the install directory; never let the uninstaller wipe it blind** (issue #199): Two independent bugs combined to destroy user data on Windows. (1) `QFileDialog.getSaveFileName`/`getOpenFileName` called with `""` as the *directory* argument open in the process **current working directory** â€” for a packaged build that is `C:\Program Files (x86)\Open Garden Planner`, so users naturally saved `.ogp` plans inside the install folder. (2) The NSIS uninstaller ran `RMDir /r "$INSTDIR"` and the upgrade path (`.onInit`) silently invokes that *old* uninstaller before installing the new build â€” so every "update" deleted the whole install dir, user plans included. Symptom: "all my gardens vanished after updating." Fixes, both required: app side â€” `GardenPlannerApp._default_dialog_dir()` resolves every save/open/export dialog to the open project's folder or `<Documents>/Open Garden Planner` (`app/paths.get_projects_dir()`), never CWD; installer side â€” the uninstall section copies any top-level `$INSTDIR\*.ogp` to `$DOCUMENTS\Open Garden Planner\Recovered Plans` *before* `RMDir /r`, so even pre-existing victims keep their data on the next upgrade. Lesson: user data must never live under `$INSTDIR`, and an installer must treat `$INSTDIR` as possibly containing user files. Regression tests: `tests/unit/test_paths.py`, `tests/integration/test_save_location.py` (the NSIS half is Windows-only, verified manually).

- **Undo/redo must dirty the document â€” and so must every "already-applied" edit; route them all through one chokepoint** (issue #209): `CommandManager.execute()` emits `command_executed`, but `undo()`/`redo()` historically emitted only `can_undo_changed`/`can_redo_changed`. Because `mark_dirty` was wired to `command_executed` ([application.py](../../src/open_garden_planner/app/application.py)), undoing or redoing after a save left the title bar's `*` marker (and the close-time "unsaved changes" prompt) believing the document was clean â€” closing then **silently discarded** the change. Repro: draw â†’ save â†’ Ctrl+Z â†’ title shows no `*`, close prompts nothing. Fix: a dedicated `stack_changed` signal on `CommandManager` (emitted by `execute`/`undo`/`redo`, **not** `clear()` â€” clearing happens on new/load and must stay clean) wired to `mark_dirty`. A dedicated signal, not "also emit `command_executed` from undo/redo", because the constraints/properties/dimension-line refreshes are *already* separately wired to `can_undo_changed`/`can_redo_changed` (firing `command_executed` too would double-fire them) and the calendar refresh would newly fire.

  **The trap that made this a two-part fix:** `command_executed` was *load-bearing* for dirtiness far beyond `execute()`. ~30 interactive-edit sites across `canvas_view.py`, `properties_panel.py`, `garden_item.py`, and the item modules (`resize_handle.py`, `circle_item.py`, `rectangle_item.py`, `ellipse_item.py`, `polygon_item.py`, `polyline_item.py`) use the "move already applied, just register undo without re-executing" pattern: they reached into `cm._undo_stack.append(cmd)` directly and hand-emitted `command_executed` to mark dirty. Moving `mark_dirty` to `stack_changed` un-dirtied every one of them (drag a resize handle / vertex / edit a radius â†’ no `*`, close discards). Naively the new integration test still passed because it only exercised the `RectangleTool` â†’ `execute()` path. Real fix: a single `CommandManager.register_applied(command)` chokepoint that does the append + redo-clear + **all** signal emits (including `command_executed` *and* `stack_changed`); every former direct-append site now calls it. Lesson: there must be exactly **two** ways onto the undo stack â€” `execute()` (runs the command) and `register_applied()` (command already applied) â€” and both emit the full signal set. Never hand-roll `_undo_stack.append` at a call site; the moment you change what `execute()` emits you create silent contract drift with every copy. Regression tests: `tests/integration/test_undo_redo_dirty.py` covers both an `execute()` path *and* a `register_applied` resize path.

- **Every live-editing input widget in the Properties panel must be in `set_selected_items`' focus guard** (issue #200): `PropertiesPanel.set_selected_items` rebuilds the entire form (`_clear_form()` â†’ `removeRow` everything â†’ recreate). It is called for *any* command via `command_executed`/`can_undo_changed`/`can_redo_changed` ([application.py](../../src/open_garden_planner/app/application.py) wires all three to a deferred `_update_properties_panel`). Live-editing fields that fire on every keystroke â€” the **Name** `QLineEdit` (`textChanged` â†’ emits `can_undo/redo_changed`) and the **Text Content** `QTextEdit` â€” therefore trigger a full rebuild *while the user is typing in them*, destroying the focused widget and dropping focus + caret after every character. Symptom: "the plant name field loses focus after each keystroke." A guard existed but whitelisted only `QDoubleSpinBox`; the text fields were overlooked. Fix: the guard returns early when `QApplication.focusWidget()` is a descendant of the panel **and** an editable input â€” `isinstance(fw, (QAbstractSpinBox, QLineEdit, QTextEdit))`. Lesson: any new field added to this panel that emits an undo/redo signal mid-edit (i.e. `textChanged`/`valueChanged` rather than `editingFinished`) must be a type the guard covers, or it will lose focus on every keystroke. Regression tests: `tests/ui/test_properties_panel.py::TestFocusPreservation` (focused field survives rebuild via a `monkeypatch` of `QApplication.focusWidget`; unfocused panel still rebuilds).

- **Free-text property fields commit on a debounce, not per keystroke â€” and not on focus-out alone** (issue #210): the Properties-panel **Name** (`QLineEdit`) and **Text Content** (`QTextEdit`) fields originally registered a `ChangePropertyCommand` on every `textChanged`. Typing an N-char value pushed **N** undo entries (Ctrl+Z walked back one character at a time) and â€” once #209 routed these through `register_applied` â†’ `command_executed` â€” fired **N** heavyweight `calendar_view.refresh()` calls (propagation plans + Gantt + dashboard + `weather_widget.set_location()`), thrashing the calendar tab once per keystroke when a GPS location was set. Fix: split **live-apply** from **commit**. `textChanged` updates the item live for visual feedback only (no command, guarded by `_updating`) and (re)starts a single-shot **debounce timer** (`_TEXT_COMMIT_DEBOUNCE_MS`, stored on the widget as `_ogp_commit_timer`); the single command is committed when the timer fires (typing pause) **or** on focus-out (`editingFinished` / the `FocusOutTextEdit` subclass's `editing_finished`, which adds the signal QTextEdit lacks), whichever is first. The start value captured at field-build time (`_ogp_name_start` / `_ogp_content_start`) is compared on commit, so an unchanged commit â€” and the debounce-then-focus-out double fire â€” push nothing. **Why the debounce and not focus-out alone** (manual-test finding): the Editâ†’Undo `QAction` (Ctrl+Z, `WindowShortcut`) is `setEnabled(False)` until `can_undo`, and shadows the focused QLineEdit's own undo. With a focus-out-only commit, pressing Ctrl+Z *while still editing* found an empty stack â†’ the action was disabled â†’ Ctrl+Z was a silent no-op and the live-applied value "stayed." The debounce makes a command exist shortly after the user stops typing, so mid-edit Ctrl+Z works. **The debounce introduces a second trap â€” flush before rebuild**: `set_selected_items` rebuilds the whole form (`_clear_form` â†’ `removeRow`), which deletes the field **and its child debounce timer synchronously**. If a debounce is still armed when the selection changes to another item (focus already off the field, so the #200 guard does not block the rebuild), the timer is destroyed before it fires â€” the edit was live-applied to the model but never recorded as a command, silently lost from undo. Fix: every field with a debounce registers its commit callback in `self._pending_text_commits` (via `_attach_commit_timer`), and `set_selected_items` calls `_flush_pending_text_commits()` *before* the rebuild, while the widgets are still alive. Each commit is idempotent (no-op when unchanged) so flushing on every rebuild is safe. Lesson: any new free-text field on this panel must follow the live-apply-on-`textChanged` + debounce/focus-out-commit pattern (never one command per keystroke, never focus-out-only), **and** route its commit through the flush queue so a cross-item rebuild cannot eat a pending edit. Regression tests: `tests/integration/test_properties_text_edit_undo.py` (`test_debounce_commits_without_focus_out`, `test_pending_edit_flushed_when_selection_changes`).

- **`QSettings.setDefaultFormat()` / `setPath()` are process-global statics Qt never reverts â€” never call them from a test** (test-suite isolation): a test fixture in `tests/unit/test_ui_state.py` redirected `UiStateStore`'s settings to a temp INI file by calling `QSettings.setDefaultFormat(IniFormat)` + `QSettings.setPath(IniFormat, UserScope, tmp_path)`. Those are **static, process-wide** and are *not* auto-reverted (the `monkeypatch` arg was declared but unused). After that unit test ran (~â…” through the suite), the global default format pointed at a `tmp_path` pytest had since deleted, so **every** `QSettings`/`AppSettings` constructed *later in the session* â€” the `get_settings()` singleton behind the snap-toggle handlers and the welcome dialog â€” wrote to a dead path and every getter fell back to its coded default. Symptom: 6 settings-persistence tests (`test_anchor_snap`, `test_nearest_snap_workflow`, `test_perpendicular_snap`, `test_tangent_snap`, `test_welcome_dialog`) **fail only in the full run, pass in isolation, and CI stays green** (different collection order/platform dodges the poison). The uniform tell: every failing getter returns exactly the *default* value, i.e. the setter write never persisted. Fix: isolate `UiStateStore` by `monkeypatch.setattr("open_garden_planner.app.ui_state.QSettings", <factory returning QSettings(ini_path, IniFormat)>)` (auto-reverts, touches no global state) â€” never the global statics. Defensive backstop in [`tests/conftest.py`](../../tests/conftest.py): the session `isolate_qsettings` fixture now captures `QSettings.defaultFormat()` and restores it at teardown (a tripwire for a future `setDefaultFormat` leak â€” it does **not** cover a `setPath`-only leak), and a function-scoped autouse `_reset_app_settings` clears the `cofade_test` store + nulls the `_settings_instance` singleton around every test so per-test value pollution can't accumulate either. Lesson: a green CI does not prove the local full suite is clean; a "passes alone, fails together" failure whose actual value is always the default is almost always shared global/singleton state, not the code under test. Deeper smell (follow-up): `UiStateStore` and `AppSettings` each construct `QSettings("cofade", â€¦)` inline, so there is no single chokepoint to isolate â€” a shared `_qsettings()` factory used by both would let one fixture isolate everything.

- **Command-description `{property}` fragments must be registered under the "Commands" i18n context** (issue #210): `ChangePropertyCommand.description` renders `translate("Commands", "Change {property}").format(property=translate("Commands", self._property_name))`. The German template is `"{property} Ã¤ndern"`, so the *fragment* must itself be a translated German noun or the undo menu reads half-English ("text content Ã¤ndern"). Every distinct `property_name` passed to `ChangePropertyCommand` (`"name"`, `"stroke width"`, `"fill color"`, `"text content"`, â€¦) is registered in [`scripts/fill_translations.py`](../../scripts/fill_translations.py) under `"Commands"`. Because all call sites pass an `apply_func`, `property_name` is purely a description fragment (never the attribute being set), so it is safe to use clean human-readable labels â€” the two ugly raw attribute names (`frost_protection_needed`, `spacing_radius_cm`) and the composed `f"text {prop}"` were normalized to nouns via `_TEXT_PROPERTY_LABELS`. An unregistered fragment degrades to English (Qt `translate` returns the source) rather than crashing â€” and `test_german_ts_has_no_unfinished` cannot catch the miss, so adding a new `ChangePropertyCommand` fragment requires manually adding its German to the `"Commands"` dict.
