## 11.4 Known Development Pitfalls

Hard-won lessons from implementation. Read these before modifying the related subsystems.

- **At a trust boundary that ingests untrusted files, catch `Exception` at the ingestion seam вЂ” do NOT enumerate exception families** (US-C4 smart symbols, PR #236, found over three senior-review rounds). The smart-symbol library loads user-dropped JSON (`<app-data>/smart_symbols/*.json`). The contract вЂ” "a bad user file must never crash the app" вЂ” was first enforced by catching `(OSError, JSONDecodeError, ValueError)`. That broke three times in a row, each a different exception family escaping the too-narrow catch: (1) `from_dict`'s dry-run `generate()` raised `ArithmeticError`/`TypeError` (divide-by-zero, `9**9**9` overflow, `round(x, float)`) вЂ” none subclass `ValueError`; (2) a deeply-nested expression (`"+".join(["1"]*5000)`) raised `RecursionError` (a `RuntimeError`) *during `ast.parse`*; (3) a non-dict `parameters` entry (`[42]`) raised `AttributeError` in `from_dict`'s **structural** phase, which runs *before* the dry-run and so bypassed `generate()`'s wrapper entirely. Each "fix" added the family that bit it, then re-declared victory вЂ” the losing game. The real fix is structural: the **user-file load loop catches `Exception`** (log-and-skip), so *any* failure of `from_dict` on untrusted input is non-fatal **by construction** (`smart_symbol_library.py`); the **bundled** load loop stays narrow (`ValueError`/`OSError`/`JSONDecodeError` в†’ crash-loud) so our own packaging bugs still surface and stay tested. `BaseException` (Ctrl-C/`SystemExit`) deliberately still propagates. Defense in depth: `generate()` also wraps its body in `except Exception в†’ SmartSymbolError` (for the *runtime* regeneration path вЂ” param edits, deserialization вЂ” which doesn't go through the loader), the evaluator caps AST node count (`parametric_eval._MAX_NODES`) so its recursion surface is bounded, and `from_dict` guards `isinstance(data, dict)`. Rule: when code ingests untrusted data, put **one broad catch at the ingestion seam**; keep narrow typed catches only for trusted/first-party input (where a surprise exception *should* be loud). Enumerating exception families at a trust boundary is a recurring bug, not a safety measure. Pinned by `tests/unit/test_smart_symbol_library.py` (arithmetic/recursion/structural/non-object poison all skipped; malformed *bundled* still crashes loud).

- **Closed (#228): task status now has a single write-path** (US-C2, #188, ADR-029): "is this task done" was originally persisted in **two** stores kept in sync at every chokepoint вЂ” the legacy `task_completions` set (read by the calendar dashboard) and the new `task_states` dict (read by the Tasks tab) вЂ” with the standing invariant that any new write path had to touch both or the surfaces silently diverged. The convergence (#228) retired that: `set_task_completion` now delegates to `set_task_status` (the single source of truth), and the calendar dashboard reads `effective_status` instead of `task_completions`, so the latter is a **write-only serialized mirror** kept only for `.ogp` forward/back-compat (older binaries still read it). There is now exactly one write-path; no future edit needs to remember the dual-write. Pinned by `test_tasks.py::TestStatusFlows::test_calendar_done_toggle_syncs_task_states` (mirror still maintained) and `test_calendar_task_convergence.py` (both surfaces agree). **The remaining caution:** any surface that reads the shared status store must still derive the `task_id` through the shared `make_calendar_task_id()` + canonical `species_key()` (the ADR-029 PR #227 addendum invariant) вЂ” convergence didn't change that.

- **Sidebar list panels must store item *ids*, not live `QGraphicsItem` refs, and defer the selection** (issue #212): `PlantSearchPanel` ("Find Plants") used to cache the live `QGraphicsItem` in each row's `UserRole` and call `setSelected(True)` on it inline. Two problems compounded into "selection works only sometimes": (1) the cached reference goes stale after undo/redo/delete/reparent вЂ” `setSelected` then silently no-ops on a detached item; (2) `QGraphicsScene.changed` (very chatty вЂ” fires on every repaint, including the repaint caused by selecting) was wired straight to a full `refresh_plant_list()` that `clear()`s and rebuilds every row, so a refresh could destroy the row the user was mid-clicking. Fix: store only the stable `item_id`, resolve the live item at click time via `scene.find_item_by_id(item_id)` (returns `None` в†’ refresh the list and bail, no crash); defer the scene mutation with `QTimer.singleShot(0, ...)` so it runs after the `QListWidget` finishes its own click processing (same pattern as `_on_companion_highlight_species`); and debounce the `scene.changed` в†’ refresh through a single-shot `QTimer` (~150 ms) so the list isn't torn down on every repaint. Single-click uses `view.ensureVisible()` (scroll only if off-screen); double-click uses `view.centerOn()`. General rule: any sidebar list that drives canvas selection must key on ids + defer, never hold graphics-item references across scene mutations.

  **Follow-up (#212 manual-test round 2): debouncing `scene.changed` is NOT enough вЂ” the refresh must also skip the rebuild when nothing changed.** The 150 ms debounce only *coalesces* a burst; it still fires once the burst settles and still did a destructive `results_list.clear()` + recreate-every-row. Because *selecting* a plant triggers its own burst (`selectionChanged` в†’ `_update_spacing_overlaps`/`_update_companion_highlights` mutate item visuals в†’ repaint в†’ more `scene.changed`), every selection ended in a full list rebuild вЂ” which (a) reset the list's scroll position (symptom: "list jumps back to the top a moment after I stop scrolling" with >5 plants) and (b) destroyed the row the user just clicked, racing the deferred `_apply_selection` (symptom: "single-click sometimes needs a second/third click"). Fix: `_update_results_display` computes a **signature** of the visible rows (ordered tuple of `(item_id, name, species, plant_type)` after sort+search+type-filter) and **early-returns when it equals the last rendered signature** вЂ” so a no-op refresh touches neither scroll nor selection. On a real rebuild it captures+restores the scroll value and re-selects the prior row by `item_id` (scroll preserved on scene-driven refreshes; reset to top on user search/filter via a `from_user_input` flag). `family` is excluded from the signature (search-only, never displayed; any family change that alters results already changes membership). Lesson: a debounce limits *how often* you rebuild, not *whether* you needed to вЂ” a list driven by the repaint firehose must also be **idempotent** (diff before rebuild) or it will fight the user's scroll and clicks. (The underlying "rebuild the world on every repaint" smell is the same one tracked for the properties panel in #206; the signature guard is a local band-aid, not a structural fix вЂ” the firehose still re-queries the whole scene every tick.) Also hardened: row widgets installed via `setItemWidget` set `WA_TransparentForMouseEvents` so a press always hit-tests to the list row regardless of the row widget's child layout.

- **Writing `metadata["plant_species"]` does not repaint the canvas, and a manual `spacing_radius_cm` override masks the database value** (issue #213): the spacing circle's radius comes from `effective_spacing_radius()`, which returns the user override *first* and only falls back to `max_spread_cm/2` from `metadata["plant_species"]`. So a direct `item.metadata["plant_species"] = вЂ¦` (as several `plant_database_panel` paths did) updates the data model but (a) never triggers `prepareGeometryChange()`/`update()`, so the spacing circle keeps its old size on screen, and (b) is silently overridden on screen by any manual spacing the user set earlier. Contract: route species *assignment* through `PlantDatabasePanel._apply_species_to_item()`, which builds a single undoable `ApplySpeciesCommand` (sets metadata **and**, on *Apply database values*, resizes the drawn footprint to `max_spread_cm` via `CircleItem.set_radius_centered()`, then repaints). The footprint resize is **silent**; the only thing that prompts is a manual `spacing_radius_cm` override that *conflicts* with the database value (Apply в†’ resize + clear override; Keep в†’ leave both). Do **not** wrap live per-keystroke field edits (`_on_field_changed`) in that command вЂ” that recreates the per-keystroke undo-churn fixed in #210; there, write metadata directly and just call `prepareGeometryChange()`/`update()` for the repaint.

- **A `scene.changed` slot that touches a `QTimer` must guard against the C++ object being torn down** (#230 CI crash): handlers wired to `QGraphicsScene.changed` that call `self._some_timer.start()` (the soil-mismatch / companion / spacing debouncers) can fire **during widget/app teardown** вЂ” the scene still emits `changed` as it is cleared, after the owning view's child `QTimer` has been deleted. Accessing the deleted timer raises `RuntimeError: wrapped C/C++ object of type QTimer has been deleted` *inside a Qt slot*, which Qt/pytest-qt escalates to `Fatal Python error: Aborted` вЂ” aborting the whole interpreter (locally it was timing-dependent and passed; on the CI Linux runner it reliably crashed the Test job). Symptom: a test that builds the full `GardenPlannerApp` and mutates the scene passes, then the *next* test errors and the run aborts mid-suite. Fix/contract: every `scene.changed` slot that starts a timer must wrap the `.start()` in `with contextlib.suppress(RuntimeError):` (see `canvas_view._on_scene_changed_for_soil`, `application._on_scene_changed_for_{companion,spacing,plant_search}`). Timers driven by *command/status* signals (`stack_changed`, `task_states_changed`) don't need it вЂ” those signals aren't emitted during teardown. **Never connect `scene.changed` to a bare `lambda: self._timer.start()`** вЂ” a lambda can't be made teardown-safe.

- **A slot connected to an *application*-global signal (`QApplication.focusChanged`) outlives its widget and must be teardown-safe** (US-C1 PR #235, surfaced merging master's `global_search` into the new full-app harvest tests): `GlobalSearchField` connects `app.focusChanged в†’ self._on_focus_changed` (the popup is a `Qt.ToolTip` that must close when focus leaves) and tried to clean up with `self.destroyed.connect(lambda _: app.focusChanged.disconnect(self._on_focus_changed))`. Two teardown faults: (1) Qt **already** auto-disconnects a destroyed receiver, so the explicit `disconnect` raced to `TypeError: 'method' object is not connected`; (2) because the connection is to the **process-global** `QApplication` (one instance across the whole test session, unlike a per-window signal), a stray `focusChanged` could still invoke `_on_focus_changed` on a field whose C++ object was gone, raising from inside the slot. pytest-qt captures any exception raised in a Qt slot during a test and **fails that test** вЂ” so the leak from one test's torn-down window failed an unrelated later full-app test (`test_harvest.py::TestHarvestNavigation`), "passes alone, fails in the full run". Fix: wrap the disconnect in `contextlib.suppress(TypeError, RuntimeError)` and guard `_on_focus_changed`'s body in `try/except RuntimeError: return`. Rule: any handler bound to an app-global Qt signal (`focusChanged`, `applicationStateChanged`, `aboutToQuit`, вЂ¦) must assume it can fire after its widget is destroyed вЂ” make both the disconnect and the slot body teardown-safe; a bare `destroyed`-lambda disconnect is not enough.

- **An enum `QComboBox` that excludes `UNKNOWN` misrepresents missing data as its first option** (issue #231): the `PlantDatabasePanel` characteristic combos (cycle / flower type / pollination / sun / water) were populated with the `UNKNOWN` member skipped (`if item != Enum.UNKNOWN`), and the populate code fell back to `setCurrentIndex(0)` when `findData(value)` missed. So a species whose trait is `UNKNOWN`/missing вЂ” common for sparse **online-search** results, where the API clients leave the enum `UNKNOWN` вЂ” was silently shown as the *first concrete option* (a tree's lifecycle read "Annual", flower type "Hermaphrodite", etc.). Worse, the save path reads `combo.currentData()` back into the model, so merely viewing then touching the form **rewrote `UNKNOWN` to that wrong concrete value** (silent data corruption). The bundled DB masked it (its records carry full trait data). Fix: prepend a neutral `(self.tr("вЂ”"), Enum.UNKNOWN.value)` entry as index 0 (mirroring the existing nutrient-demand combos), so unknown renders as "вЂ”" and round-trips as `UNKNOWN`. Rule: any combo bound to an enum with an `UNKNOWN`/unset member must include a neutral entry for it вЂ” never let the index-0 fallback assert a specific value. Pinned by `tests/ui/test_plant_database_panel.py`. **Sibling defect, same issue (now fixed):** the dimension/pH/hardiness spin-boxes called `setSpecialValueText("")` вЂ” but Qt treats an **empty** special-value string as *disabling* the feature, so the unset sentinel (`value == minimum == 0`, the model's `0 в†” None` "unknown") rendered as a concrete "0 cm"/"0"/"0.0". A **non-empty** `setSpecialValueText(self.tr("вЂ”"))` engages the placeholder (mirrors `properties_panel.py`, `soil_test_dialog.py`, `location_dialog.py`). Purely a display change вЂ” populate still does `setValue(x or 0)` and save still reads `value() if > 0 else None`, so `.value()` returns `0` at the dash and the `None` round-trip is unchanged. Rule: a spin-box placeholder needs a **non-empty** special-value text paired with `minimum` as the sentinel; `setSpecialValueText("")` is a silent no-op.

  **Manual-test follow-up (#213): refreshing only the spacing ring was not enough.** The spacing ring is only painted when the item is selected **and** `spacing_r > footprint_radius` (`circle_item.py`); when a generic plant is placed at a footprint larger than the assigned species' spread (the common case вЂ” a 5 m placeholder getting a 1.5 m shrub), the ring falls *inside* the footprint and is never drawn, so "nothing resizes on screen" despite the panel showing the new values. The issue explicitly lists **Diameter** among the properties that must update, so *Apply database values* now resizes the **drawn footprint** itself (diameter == `max_spread_cm`, centre fixed) rather than only the ring. Lesson: the visible artefact a user expects to change is the footprint circle, not the (often hidden) spacing ring вЂ” reconcile the thing on screen.

  **Second trap (senior review): programmatic resize must keep the rotation pivot on the centre.** A `CircleItem`'s saved position is `pos + center` (see `core/project.py` serialization), with rotation stored as a *separate* angle pivoting on the centre вЂ” i.e. the code relies on the invariant `transformOriginPoint == center == rect.center()`. A resize that only keeps the *visual* centre fixed (via `mapToScene` compensation) but forgets to move `transformOriginPoint` to the new centre will save a **displaced** `center_x/center_y` for any **rotated** plant (and make the next rotation jump). It looks correct on screen immediately, so an unrotated test passes green. `CircleItem.set_radius_centered()` therefore calls `setTransformOriginPoint(new_center)` and repositions so `pos + center` stays put; `test_apply_keeps_rotated_plant_centered` pins it. General rule for this codebase: any geometry mutation on a rotatable item must preserve `transformOriginPoint == rect.center()`, and resize-under-rotation needs an explicit rotated-item test.

  **Closed (#219): rotation now pivots about `rect().center()`, not `boundingRect().center()`.** The latent half of the trap above lived in `RotationHandleMixin._apply_rotation()` (`resize_handle.py`), which set the pivot to `boundingRect().center()`. For a plant showing the **antagonist warning badge** that bounding box is expanded *asymmetrically* (`circle_item.py` `boundingRect()` adds `+x/+y` overflow only), so the pivot was off `rect().center()` вЂ” breaking the serializer invariant *before any resize*. Because `_antagonist_warning` is **runtime-only (never serialized)**, the save-time pivot (badge on в†’ off-centre) disagreed with the load-time pivot (badge off в†’ centred): a rotated badged plant drifted on save/reload with no resize involved. Fix: pivot about `rect().center()` when the item exposes `rect()` (`CircleItem`/`RectangleItem`/`EllipseItem`), else fall back to `boundingRect().center()` (`TextItem`/`CalloutItem`/`PolylineItem`, which have no asymmetric decoration). This makes the pivot invariant to badge state, so toggling the badge needs no re-rotation. Pinned by `test_rotation_pivots_about_rect_center_with_badge`. Lesson: a rotation pivot must come from the *geometric* shape (`rect()`), never the decoration-expanded `boundingRect()` вЂ” runtime-only decorations otherwise leak into persisted geometry.

  **Closed (#218): the interactive resize of a *rotated* item was geometrically incoherent вЂ” fix the geometry step, don't post-correct it.** Symptom (PR #221 manual test): drag-resizing a 45В°-rotated plant made the diameter collapse / refuse to grow, the centre drifted, the dragged handle didn't track the cursor, and a stale "ghost" disc lingered. Three compounding causes: (a) `CircleItem._apply_resize` squared via `min(width, height)` вЂ” at 45В° a diagonal drag projects onto **one** local axis (`local_dyв‰€0`), so `min` picked the *unchanged* axis and the circle could not grow at all (a headless trace showed radius stuck at 50 through an (80,80) drag); a MIDDLE handle (`new_height == init_height`) was capped likewise. (b) `CircleItem._apply_resize` inferred the fixed edge from **scene-space** `abs(pos_x в€’ init_pos.x()) < 0.01` while the first-cut re-anchor inferred it from the **rotated local** `pos_dx == 0` вЂ” the two disagreed under rotation, so the re-anchor pinned the wrong corner and the centre drifted (45вЂ“135 cm in the trace). (c) neither `_apply_resize` nor the helper called `prepareGeometryChange()`, so a shrinking `boundingRect()` (which includes the spacing-ring expansion) left stale pixels. Fix: the interactive `ResizeHandle._apply_resize` now takes the fixed corner/edge **authoritatively from `self._position`**, computes the new local rect, lets the item normalise it (`CircleItem._constrain_resize_size` squares it so the dragged handle *tracks the cursor* вЂ” corner в†’ `max(w,h)`, edge в†’ that axis, which also lets a side handle grow a circle), applies it through `resize_rect_item_keeping_anchor` (which now `prepareGeometryChange()`s and re-pins the origin), and refreshes via `_after_resize_geometry()`. The earlier rotation-gated `_reanchor_after_rotated_resize` band-aid and the `min(w,h)` + scene-space guess are gone. Scope: rect-bearing items (`Circle`/`Rectangle`/`Ellipse`); polygons keep the axis-aligned path (`hasattr 'rect'` gate). The programmatic `_apply_resize(x,y,w,h,pos,pos)` stays for equal-constraint partner resizes. Pinned by `tests/integration/test_rotation_aware_resize.py` ({shape}Г—{0,45,215В°}Г—{corner,edge}). Lessons: **(1)** every geometry mutation on a rotatable item must end with `transformOriginPoint == rect().center()`, centralised in one helper; **(2)** don't *post-correct* an incoherent geometry step вЂ” fix the step (a band-aid over `min(w,h)` + dual fixed-corner inference can't be made correct); **(3)** any code that shrinks a custom `boundingRect()` must `prepareGeometryChange()` or Qt leaves ghosts.

  **Plant sizing precedence now has one home (#218): `core/plant_sizing.py`.** The three sizing quantities (drawn footprint `_radius`, manual `spacing_radius_cm` override, DB `max_spread_cm`) and their precedence (override > `max_spread`/2 > None; ring shown only when effective > footprint) used to be re-encoded inline in `garden_item.effective_spacing_radius()`, `circle_item.paint()`/`boundingRect()`, and `plant_species_assignment.py`. They now all delegate to the Qt-free `PlantSizing` value object / `sizing_for_item` / `db_spacing_radius_cm` resolver. Behaviour is unchanged (pinned by `tests/unit/test_plant_sizing.py` + the existing `test_spacing_circles.py`); the win is that the next person changing "what size is this plant" edits one file. The underlying "rebuild on every repaint" overlap-refresh smell (#206) is untouched.

- **Release workflow race condition with chore commits**: After merging a feature PR, two chore commits are pushed (sync version + mark progress). These land ~37s after the PR merge but the Release workflow building the new tag takes ~2m50s. The chore-commit Release runs start while the tag doesn't exist yet, compute a stale version (e.g., `v1.8.4` instead of `v1.9.2`), and fail with "release with the same tag name already exists". Fixed by adding `if: "!startsWith(github.event.head_commit.message, 'chore:')"` to the release job, which skips the workflow for chore commits.

- **Anchor index on same-type anchors**: When multiple anchors share the same `AnchorType` (e.g. rectangle corners are all `CORNER`, polygon vertices are all `CORNER`, polyline vertices are all `ENDPOINT`), each must have a unique `anchor_index` in `get_anchor_points()`. Without it, `DimensionLineManager._resolve_anchor_position()` falls back to type-only matching and picks the first anchor. Always pass `anchor_index=i` when creating `AnchorPoint` for same-type anchors.

- **Dimension line updates after undo/redo**: `CommandManager.command_executed` only fires on `execute()`, NOT on `undo()`/`redo()`. Dimension line updates must also be connected to `can_undo_changed`/`can_redo_changed` signals.

- **3-anchor constraints not solved on add**: `_compute_constraint_solve_moves()` in `canvas_view.py` collects `constrained_ids` from `anchor_a` and `anchor_b` only. Any constraint with a third anchor (`anchor_c`, e.g. ANGLE) must also add `anchor_c.item_id` here, otherwise the third item is absent from `item_positions` and the solver cannot move it вЂ” showing as red/violated until the user manually drags an object.

- **Live-drag projection tolerance must be sub-pixel** (PR #169 follow-up): `ConstraintGraph.project_to_feasible()` calls `newton_refine()` with a `tol` parameter that gates an early-return *before* projection: `if max_err <= tol: return`. The starting state has `vertex_pos[moving] = desired_scene_pos` (the cursor). At cm-scale `tol` (e.g. 0.5 cm), the cursor near a feasible point falls inside the slack band on most frames of a near-stationary drag, so the function returns the cursor **unchanged** вЂ” the moving vertex slips with the cursor up to that band each frame. On a fully-constrained chain (EDGE_LENGTH on every adjacent pair), the slip looks like the entire polyline rigidly translating because the user can only see the moving vertex's position in scene coords. Fix: `project_to_feasible` defaults to `tolerance=1e-4` (sub-render-precision); `solve_anchored` and other full-graph callers retain their cm-scale tolerances. Defensive companion: `newton_refine` now also calls `write_x(x)` on the early-return path so callers always observe a consistent state.

- **Canvas Y-axis flip**: The view applies `scale(zoom, -zoom)` so **positive scene Y is visually upward** on canvas (CAD-style, origin bottom-left). When computing directional offsets from user-facing angles (e.g. linear array), negate `dy`: `dy = -spacing * sin(angle_rad)` so that 0В°=right, 90В°=down, 180В°=left, 270В°=up matches screen-space intuition. The canvas rect in scene coords is `QRectF(0, 0, width_cm, height_cm)` accessed via `self._canvas_scene.canvas_rect`.

- **Agent API described the compass backwards (issue #267, fixed)**: every Agent API tool description (move/read/spatial/render) called the scene frame "origin top-left, +y down" вЂ” Qt's *default* convention, but wrong for OGP, which is CAD Y-up (ADR-002; see the *Canvas Y-axis flip* entry above). Because the scene renders north-up, an agent told "move the bed 7 m south" computed `dy = +700` and the bed moved **north**. The *math* was correct and internally consistent (read + move + spatial all use raw scene y); only the descriptions lied. Fixed by rewriting every agent-facing description to state **origin bottom-left, +x east, +y north (a larger y is further north); to move south, pass a negative dy**. NOT fixed by negating `dy` in the tool вЂ” that would desync `move_object` from the read tools it must stay consistent with. The render pixel formula (`px_y = image_height_px - (y_cm - region_y_cm) * px_per_cm`) was already correct (Y-up image, top-down pixel rows) and test-pinned; only its prose was corrected. Regression-guarded by `tests/unit/test_agent_api_auth.py::test_move_object_description_uses_y_up_compass_frame` (the description must say north/south, never "+y down").

- **EDGE_* anchor type instability on polygons/polylines** (TD-007): `_polygon_anchors()` in `measure_snapper.py` classifies each edge as `EDGE_TOP/BOTTOM/LEFT/RIGHT` based on its **current** dominant axis (horizontal vs vertical). This is a dynamic, geometry-dependent label. When a vertex is dragged far enough to flip an edge's dominant axis (e.g. a nearly-horizontal edge becomes nearly-vertical), the freshly-computed anchor type differs from the value stored in the constraint record. `_resolve_anchor_position()` in `dimension_lines.py` then fails on its `(type AND index)` match and falls through to type-only matching, snapping the constraint indicator to the wrong edge. Current workaround: an EDGE_*-aware index-only match block at the top of `_resolve_anchor_position` catches these mismatches before the fallback fires. Long-term fix (TD-007): replace all four `EDGE_*` types with a single `AnchorType.EDGE_MIDPOINT` and use `anchor_index` as the sole edge identity. The edge is always identified by its start-vertex index, not by axis classification, making the anchor stable across all vertex moves.

Repro: draw в†’ save в†’ Ctrl+Z в†’ title shows no `*`, close prompts nothing. Fix: a dedicated `stack_changed` signal on `CommandManager` (emitted by `execute`/`undo`/`redo`, **not** `clear()` вЂ” clearing happens on new/load and must stay clean) wired to `mark_dirty`. A dedicated signal, not "also emit `command_executed` from undo/redo", because the constraints/properties/dimension-line refreshes are *already* separately wired to `can_undo_changed`/`can_redo_changed` (firing `command_executed` too would double-fire them) and the calendar refresh would newly fire.

  **The trap that made this a two-part fix:** `command_executed` was *load-bearing* for dirtiness far beyond `execute()`. ~30 interactive-edit sites across `canvas_view.py`, `properties_panel.py`, `garden_item.py`, and the item modules (`resize_handle.py`, `circle_item.py`, `rectangle_item.py`, `ellipse_item.py`, `polygon_item.py`, `polyline_item.py`) use the "move already applied, just register undo without re-executing" pattern: they reached into `cm._undo_stack.append(cmd)` directly and hand-emitted `command_executed` to mark dirty. Moving `mark_dirty` to `stack_changed` un-dirtied every one of them (drag a resize handle / vertex / edit a radius в†’ no `*`, close discards). Naively the new integration test still passed because it only exercised the `RectangleTool` в†’ `execute()` path. Real fix: a single `CommandManager.register_applied(command)` chokepoint that does the append + redo-clear + **all** signal emits (including `command_executed` *and* `stack_changed`); every former direct-append site now calls it. Lesson: there must be exactly **two** ways onto the undo stack вЂ” `execute()` (runs the command) and `register_applied()` (command already applied) вЂ” and both emit the full signal set. Never hand-roll `_undo_stack.append` at a call site; the moment you change what `execute()` emits you create silent contract drift with every copy. Regression tests: `tests/integration/test_undo_redo_dirty.py` covers both an `execute()` path *and* a `register_applied` resize path.

- **Properties panel updates incrementally вЂ” rebuild only on a genuine selection/structure change** (issue #206, supersedes the #200 focus-guard workaround): `PropertiesPanel.set_selected_items` used to rebuild the entire form (`_clear_form()` в†’ `removeRow` everything в†’ recreate) on *every* call, and it is called for *any* command via `command_executed`/`can_undo_changed`/`can_redo_changed` ([application.py](../../src/open_garden_planner/app/application.py) wires all three to a deferred `_update_properties_panel`). Live-editing fields that fire on every keystroke вЂ” the **Name** `QLineEdit` and **Text Content** `QTextEdit` вЂ” therefore tore down the form *while the user typed in them*, destroying the focused widget and dropping focus + caret after every character ("the plant name field loses focus after each keystroke", #200). The #200 fix was a focus guard (skip the rebuild while a panel editor holds focus), but that guard accreted whitelisted widget types and traded a stale view under focus. **#206 fixes the root cause:** `set_selected_items` now computes a structural **identity** (`_compute_identity`: item id + class + `object_type` + the bed's `child_item_ids` / a plant's `parent_bed_id`, which drive *which* fields and read-only summary rows exist, **plus** `_relationship_summary_key` вЂ” the rendered *text* of the Parent Bed / Contained Plants rows. Those rows show a **related** item's name; the id set alone does not catch a rename/undo of that related item while the current item stays selected, which would leave the summary stale on the refresh path since no refresher covers a variable-row summary list. `id(item)` is load-bearing: it is correct only because commands mutate the selected item in place and reuse the same object across undo/redo вЂ” a command that swaps the object for the same selection must add a discriminator). When the identity is unchanged вЂ” a pure scalar property edit, a canvas drag/resize, or undo/redo on the same item вЂ” it keeps the widgets alive and pushes fresh model values into them via the registered field **refreshers** (`_register_refresh` appends a closure next to each widget's creation; `_refresh_field_values` runs them all). Each refresher **skips a focused widget** (`_is_focused`) and writes under `blockSignals` so a value push never registers a command or stomps an active edit. The form is rebuilt only on an identity change (different/multi/empty selection, type change, reparent). The old focus guard at the top of `set_selected_items` is **kept as a defensive backstop** for the rare "selection genuinely changed while a field holds focus" rebuild вЂ” it is no longer load-bearing for the common case, so new live-editing fields no longer have to be whitelisted in it. **A naive "skip the rebuild when the selection is unchanged" was rejected** because it would leave Position/Size/colour showing stale values after a canvas drag or undo/redo; the refresh path is what keeps them correct. Lesson: any new editable field added to this panel must register a refresher (`_register_refresh(widget, lambda: widget.setвЂ¦(read_model()))`) so its value stays in sync without a rebuild; any new field whose presence depends on a model relationship must be folded into `_compute_identity` so that relationship change forces a rebuild. **A combo whose *item list* (not just its selected index) is backed by mutable external state must repopulate that list in its refresher, not merely re-index** (#222 senior-review P1): the **Layer** combo reads `scene.layers`, and a re-index-only refresher left a renamed layer showing its old name / omitted a newly-added layer on the refresh path вЂ” its refresher now `clear()`s + re-`_populate_layer_combo()`s under `blockSignals` (`_refresh_layer_combo`). Object-type/style/pattern combos are enum-backed (fixed list) so a re-index is sufficient there. Regression tests: `tests/integration/test_properties_panel_incremental.py` (no-rebuild-on-same-selection, value refresh after move/resize/undo/redo, rebuild on selection change, refresher skips focused widget, **read-only summary rows stay fresh on a related-item rename/undo**, **Layer combo repopulates on a layer rename/add**) and `tests/ui/test_properties_panel.py::TestFocusPreservation`. The unguarded sibling `plant_database_panel.set_selected_items` (which already uses persistent widgets) got a matching defensive `_holds_field_focus()` backstop: it skips the value re-push when called for the already-displayed item while one of its own fields holds focus.

  **Follow-up (#206 wiring): the panel refreshes are driven by a single `stack_changed` connection, not the three-signal fan-out.** #222 fixed the panel *internals* but left the `application.py` wiring as `command_executed` + `can_undo_changed` + `can_redo_changed` в†’ a deferred `_update_*` (three fan-outs per command; `constraints_panel.refresh()` actually ran up to 3Г— since it isn't `singleShot`-coalesced). They now collapse to one `cmd_mgr.stack_changed.connect(...)` per panel (`stack_changed` fires exactly once per `execute`/`register_applied`/`undo`/`redo`, never on `clear()`); `can_undo/redo_changed` stay wired **only** to the toolbar Undo/Redo action enablement (their real job). **No undo/redo coverage is lost in the swap:** the constraints and properties panels already refreshed on undo/redo via `can_undo/redo_changed` (those `pyqtSignal`s are emitted unconditionally on every `undo()`/`redo()` in `commands.py`, *not* only when the boolean flips), so moving them to `stack_changed` is behaviour-preserving for those two. The genuinely *new* fix is the **plant-database panel**: it was wired only to `selectionChanged` / `object_type_changed` (never to any command signal), so undoing/redoing a species assignment while the plant stayed selected left its species details stale until reselection вЂ” it now also hangs off `stack_changed`. Regression test: `tests/integration/test_panel_refresh_wiring.py` (one refresh per panel per execute/undo/redo, incl. the plant-DB undo/redo coverage, plus toolbar enablement intact). **Closed (#225):** the three stragglers вЂ” `calendar_view.refresh()` wired to `command_executed` only (so off undo/redo), and the companion / crop-rotation panels wired `selectionChanged`-only вЂ” are fixed. Companion + crop-rotation now also hang off `cmd_mgr.stack_changed` (cheap list rebuilds). The calendar gained a debounced `schedule_refresh()` that skips work while its tab is hidden (so the heavyweight `refresh()` that #210 flagged can safely be wired to `stack_changed` + `task_states_changed`); `_on_tab_changed` still refreshes it on show. Regression test: `tests/integration/test_panel_refresh_wiring.py::TestCompanionCropRotationRefreshOnUndoRedo`.

- **Free-text property fields commit on a debounce, not per keystroke вЂ” and not on focus-out alone** (issue #210): the Properties-panel **Name** (`QLineEdit`) and **Text Content** (`QTextEdit`) fields originally registered a `ChangePropertyCommand` on every `textChanged`. Typing an N-char value pushed **N** undo entries (Ctrl+Z walked back one character at a time) and вЂ” once #209 routed these through `register_applied` в†’ `command_executed` вЂ” fired **N** heavyweight `calendar_view.refresh()` calls (propagation plans + Gantt + dashboard + `weather_widget.set_location()`), thrashing the calendar tab once per keystroke when a GPS location was set. Fix: split **live-apply** from **commit**. `textChanged` updates the item live for visual feedback only (no command, guarded by `_updating`) and (re)starts a single-shot **debounce timer** (`_TEXT_COMMIT_DEBOUNCE_MS`, stored on the widget as `_ogp_commit_timer`); the single command is committed when the timer fires (typing pause) **or** on focus-out (`editingFinished` / the `FocusOutTextEdit` subclass's `editing_finished`, which adds the signal QTextEdit lacks), whichever is first. The start value captured at field-build time (`_ogp_name_start` / `_ogp_content_start`) is compared on commit, so an unchanged commit вЂ” and the debounce-then-focus-out double fire вЂ” push nothing. **Why the debounce and not focus-out alone** (manual-test finding): the Editв†’Undo `QAction` (Ctrl+Z, `WindowShortcut`) is `setEnabled(False)` until `can_undo`, and shadows the focused QLineEdit's own undo. With a focus-out-only commit, pressing Ctrl+Z *while still editing* found an empty stack в†’ the action was disabled в†’ Ctrl+Z was a silent no-op and the live-applied value "stayed." The debounce makes a command exist shortly after the user stops typing, so mid-edit Ctrl+Z works. **The debounce introduces a second trap вЂ” flush before rebuild**: `set_selected_items` rebuilds the whole form (`_clear_form` в†’ `removeRow`), which deletes the field **and its child debounce timer synchronously**. If a debounce is still armed when the selection changes to another item (focus already off the field, so the #200 guard does not block the rebuild), the timer is destroyed before it fires вЂ” the edit was live-applied to the model but never recorded as a command, silently lost from undo. Fix: every field with a debounce registers its commit callback in `self._pending_text_commits` (via `_attach_commit_timer`), and `set_selected_items` calls `_flush_pending_text_commits()` *before* the rebuild, while the widgets are still alive. Each commit is idempotent (no-op when unchanged) so flushing on every rebuild is safe. Lesson: any new free-text field on this panel must follow the live-apply-on-`textChanged` + debounce/focus-out-commit pattern (never one command per keystroke, never focus-out-only), **and** route its commit through the flush queue so a cross-item rebuild cannot eat a pending edit. Regression tests: `tests/integration/test_properties_text_edit_undo.py` (`test_debounce_commits_without_focus_out`, `test_pending_edit_flushed_when_selection_changes`).

- **`QSettings.setDefaultFormat()` / `setPath()` are process-global statics Qt never reverts вЂ” never call them from a test** (test-suite isolation): a test fixture in `tests/unit/test_ui_state.py` redirected `UiStateStore`'s settings to a temp INI file by calling `QSettings.setDefaultFormat(IniFormat)` + `QSettings.setPath(IniFormat, UserScope, tmp_path)`. Those are **static, process-wide** and are *not* auto-reverted (the `monkeypatch` arg was declared but unused). After that unit test ran (~в…” through the suite), the global default format pointed at a `tmp_path` pytest had since deleted, so **every** `QSettings`/`AppSettings` constructed *later in the session* вЂ” the `get_settings()` singleton behind the snap-toggle handlers and the welcome dialog вЂ” wrote to a dead path and every getter fell back to its coded default. Symptom: 6 settings-persistence tests (`test_anchor_snap`, `test_nearest_snap_workflow`, `test_perpendicular_snap`, `test_tangent_snap`, `test_welcome_dialog`) **fail only in the full run, pass in isolation, and CI stays green** (different collection order/platform dodges the poison). The uniform tell: every failing getter returns exactly the *default* value, i.e. the setter write never persisted. Fix: isolate `UiStateStore` by `monkeypatch.setattr("open_garden_planner.app.ui_state.QSettings", <factory returning QSettings(ini_path, IniFormat)>)` (auto-reverts, touches no global state) вЂ” never the global statics. Defensive backstop in [`tests/conftest.py`](../../tests/conftest.py): the session `isolate_qsettings` fixture now captures `QSettings.defaultFormat()` and restores it at teardown (a tripwire for a future `setDefaultFormat` leak вЂ” it does **not** cover a `setPath`-only leak), and a function-scoped autouse `_reset_app_settings` clears the `cofade_test` store + nulls the `_settings_instance` singleton around every test so per-test value pollution can't accumulate either. Lesson: a green CI does not prove the local full suite is clean; a "passes alone, fails together" failure whose actual value is always the default is almost always shared global/singleton state, not the code under test. Deeper smell (follow-up): `UiStateStore` and `AppSettings` each construct `QSettings("cofade", вЂ¦)` inline, so there is no single chokepoint to isolate вЂ” a shared `_qsettings()` factory used by both would let one fixture isolate everything.

- **Command-description `{property}` fragments must be registered under the "Commands" i18n context** (issue #210): `ChangePropertyCommand.description` renders `translate("Commands", "Change {property}").format(property=translate("Commands", self._property_name))`. The German template is `"{property} Г¤ndern"`, so the *fragment* must itself be a translated German noun or the undo menu reads half-English ("text content Г¤ndern"). Every distinct `property_name` passed to `ChangePropertyCommand` (`"name"`, `"stroke width"`, `"fill color"`, `"text content"`, вЂ¦) is registered in [`scripts/fill_translations.py`](../../scripts/fill_translations.py) under `"Commands"`. Because all call sites pass an `apply_func`, `property_name` is purely a description fragment (never the attribute being set), so it is safe to use clean human-readable labels вЂ” the two ugly raw attribute names (`frost_protection_needed`, `spacing_radius_cm`) and the composed `f"text {prop}"` were normalized to nouns via `_TEXT_PROPERTY_LABELS`. An unregistered fragment degrades to English (Qt `translate` returns the source) rather than crashing вЂ” and `test_german_ts_has_no_unfinished` cannot catch the miss, so adding a new `ChangePropertyCommand` fragment requires manually adding its German to the `"Commands"` dict.


- **Two surfaces that share a status store must derive the key identically вЂ” reuse the canonical `species_key()`** (US-C2, PR #227 #12): the Planting-Calendar dashboard and the Tasks tab both persist per-task done/snooze state in `task_states`, keyed by `task_id = "{species_key}:{task_type}:{year}"` for calendar-derived tasks. They computed `species_key` *differently*: the calendar used the canonical `species_key()` helper ([`models/plant_data.py`](../../src/open_garden_planner/models/plant_data.py), ADR-016 вЂ” `source_id в†’ scientific_name в†’ common_name`, **lowercased**), while `tasks_view.build_plan_state` used a raw `sp.scientific_name or sp.common_name or item.name` (no `source_id`, not lowercased). For any DB-sourced species (carries a `source_id`) or any casing difference the two ids diverged (`12345:harvest:2026` vs `Solanum Lycopersicum:harvest:2026`), so marking a task done on one surface was invisible to the other вЂ” sync failed in **both** directions. The round-2 PR test masked it by asserting the data layer with hand-matched ids. Fix: `build_plan_state` now calls the same `species_key()` helper, **and** the id *format* is built through a single shared `make_calendar_task_id()` (in `services/task_generator.py`) used by both the generator and the calendar dashboard вЂ” so neither the key nor the format can drift again. Lesson: when two views read/write a shared id-keyed store, the *entire* id derivation (key + format) must be a single shared function, and a sync test must build the id the way *production* does (from a realistic species with a `source_id`), not hand-write a matching string. Regression test: `tests/integration/test_tasks.py::TestCrossSurfaceSync`.

- **A handler fed a canonical `species_key` must match on the canonical key, not a raw name вЂ” and a "navigate by species" channel must also handle unkeyed targets** (US-C1, PR #235 manual test): the Harvest dashboard double-click routes through the shared `GardenPlannerApp._on_highlight_species`, also wired to the planting-calendar `highlight_species` and the Tasks tab `navigate_to_species` вЂ” **all three emit a canonical `species_key()`** (ADR-016, lowercased `source_id в†’ scientific в†’ common`). The handler compared that key against each item's **raw-cased** `scientific_name or common_name`, so it never matched for any capitalised name or any DB species carrying a `source_id` в†’ nothing got selected (latent on all three surfaces; surfaced by harvest). Fix: compare `species_key(item.metadata["plant_species"])` on both sides + `centerOn` the first match. Second half of the same bug: a harvest on a **bed or species-less plant** aggregates under a `target:<uuid>` grouping key (it has no species), and the view had deliberately blanked those so they navigated *nowhere*. `_on_highlight_species` now also handles a `target:<uuid>` key by selecting that one item by id (via `_select_items_by_id`), and the view stashes the full grouping key. The same unkeyed target also caused the dashboard/CSV/PDF to show the raw `target:<uuid>` as a *name*: fixed by caching the item's localized object-type name at log time (`_harvest_species_for_target`) and substituting a localized "Unnamed" at every display surface (table + CSV + PDF вЂ” all three, or they drift). Lessons: **(1)** a signal documented as carrying a `species_key` must be *consumed* with the same canonical derivation it was produced with вЂ” never re-derive from a display name; **(2)** a "select by species" channel needs an explicit branch for targets that have no species (beds), or they become silently unclickable; **(3)** an internal grouping key (`target:<uuid>`) must never leak into a user-facing name column. Regression tests: `tests/integration/test_harvest.py::TestHarvestNavigation` (plant-by-species + bed-by-id + type-name fallback), `::TestAggregationAndExport::test_csv_unnamed_target_shows_label_not_blank`.

- **A `statusBar().showMessage()` set while a modal dialog is open is invisible вЂ” and a transient status blip is the wrong channel for a startup reminder** (US-C2, PR #227 #16): the overdue-task startup reminder used `self.statusBar().showMessage(msg, 10000)`. On the common path (Welcome dialog в†’ recent project) the message was written by `_open_project_file` *while the modal `WelcomeDialog` still covered the main window* (the dialog emits `recent_project_selected` в†’ loads the project в†’ `accept()`s only afterwards), so it was painted behind the dialog and never seen; even the post-dialog re-check was a 10 s bottom-of-window blip competing with `"Opened: вЂ¦"` / `"Auto-saved"` / `"Ready"`. Net: the user never saw it, toggle on or off. Fix: deliver via a **persistent, dismissible top bar** (`TaskReminderBar`, modelled on `UpdateBar`) with a "Show Tasks" action, and run the check **deferred** (`QTimer.singleShot(0, self._check_overdue_tasks)`) from both the startup sequence and `_open_project_file` so it fires after the modal dialog has closed and the event queue has drained. The check is idempotent (re-evaluates the count, hides on zero) so multiple deferred calls are safe. Lesson: never use the status bar for a notification that must survive a modal dialog or compete with other writes вЂ” use a persistent bar; and defer any post-open UI feedback past the modal `exec()` that may have triggered it. Regression test: `tests/integration/test_tasks.py::TestOverdueReminderBar`.
